Ad Bounty — Launcher Flow

Approved creatives → the brand's Meta ad account. Two modes per brand (brand_meta_config.launch_mode): auto — ready to launch auto-creates the ads (the engine flow in Phases 1–3 below); advanced — ready to launch only queues the task for a media buyer, who assembles + pushes from the cockpit (V2 section). Both call the same engine core; either way every ad lands PAUSED and a human activates it. Click any step to expand.

Operating the Launcher — team SOPs

Media buyer — run a launch (advanced-mode brands)

Log in to the cockpit (ad-bounty-launcher-cockpit.vercel.app) — you see only the brands you're granted.
Open the brand's queue (tasks at ready to launch, oldest first) and select one or more tasks to launch together.
Assemble → review each ad: destination URL, headline / primary text / CTA (creator copy if supplied, else brand stock), and ad set (clone the brand's template, or pick an existing one). Edit anything; only edited ads are changed.
Preview, then Push. The engine creates one PAUSED ad per creative; the task flips to learning only once all its ads land.
Open Ads Manager, review the paused ads, and activate when ready. Nothing spends until you do.

Auto-mode brands

No cockpit step. Ready to launch auto-creates the PAUSED ads; the brand's media buyer just reviews and activates in Ads Manager.

Onboard a brand to the Launcher

Collect from the brand: their Ad Bounty campaign, a template ad set inside it, the Facebook page, default landing URL + CTA, and 5–10 stock copy pairs.
Access (Zak): partner share at Advertiser level + ad account assigned to the system user with ADVERTISE. The page works via the BM share — a page read failing is normal; the first paused launch is the real test.
Provision: insert the brand_meta_config row (+ launch_mode) and stock copy; for advanced mode also grant each buyer cockpit access.
Gate: validate_brand_config.py must print CONFIG QC: GREEN, then set is_active = true. Brands without an active config are silently skipped (stay on the manual flow).

When a launch fails or gets stuck

Read the ClickUp comment — every failure names the step and the fix (bad URL, missing copy, page permission, etc.). The comment trail + status tells the whole story.
Transient errors (Meta rate limits, video processing) retry automatically — no action needed.
Permanent errors: fix the root cause, then move the task back to ready to launch to re-trigger. Resume is idempotent — no duplicate ads. (Or wait for the daily reconciler.)

Quality Review

1. What breaks?

Webhook never arrives: Daily reconciler picks up anything stuck >30 min. Worst case: one day late, never lost.
Same webhook delivered twice / task re-flipped: Three idempotency layers (webhook_events PK, launch_runs idempotency_key, deterministic ad names + adopt-by-name). Proven: replay produced zero duplicates.
Worker dies mid-launch: Every Meta ID is persisted the moment it exists; resume skips completed assets. Proven via SIGKILL test.
Accidental spend: Impossible by design — every object lands PAUSED; only the brand's media buyer activates.
Wrong campaign placement: /copies clones into the source ad set's campaign; the Config QC gate asserts source_adset_id lives inside the brand's Ad Bounty campaign before enablement.

2. Who knows?

Success: ClickUp comment with ad set + ad IDs + Ads Manager link; status → learning.
Validation failure: Task reverts to approved with a 🛑 comment naming the exact problem (missing/invalid URL, no copy).
Permanent failure: ❌ comment with the failing step and the fix. The comment trail + status tells the whole story without opening other tools.
Forensics: /admin/runs endpoint + launch_runs/launch_actions/launch_media_assets give per-step history for any launch.

3. What's the fallback?

Launcher down entirely: The manual upload flow still works — nothing in the old path was removed. Tasks just sit at ready to launch.
Brand not yet onboarded: Silently skipped by design; manual flow unaffected.
Drive file moved/deleted: Download 404s → permanent-failure comment names the file.
Stuck partial run: Move the task back to ready to launch (or wait for the reconciler) — resume is idempotent.
