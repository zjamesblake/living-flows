Verification Controls — How We Guarantee Every Dollar

Five guarantees, each enforced by a specific mechanism. If a guarantee can’t be proven, the period cannot close.

GuaranteeMechanismDetail
Calculated → Owed
Sums add up correctly		 Reconciliation R1–R8 R4: brand spend = sum of student spend. R5: total spend = sum of all student spend. R6: total payout = sum of all payouts. R7: brand payout = sum of brand’s student payouts. R3: Meta totals = Atria totals. Any failure stops the skill.
Owed → Sent (students)
Wise actually moved the right amount		 Skill 8 + Wise API For every paid ClickUp task, Skill 8 pulls the Payment Reference, calls Wise API, confirms: (1) transfer exists, (2) amount within $0.50 of calculated, (3) status in {outgoing_payment_sent, funds_converted, processing}. Mismatch → status flagged, period stays open, #ops-digest alerts.
Owed → Received (brands)
Brand actually paid the invoiced amount		 Skill 8 invoice check For every brand_invoices row: status must be paid or verified, payment_amount must be set, and |payment_amount − invoice_amount| ≤ $0.50. Any miss → period stays open. (Amount-match check added 2026-05-18.)
Adjustments
Every top-up has its own ledger entry + verified Wise transfer		 /payout-adjust + option B webhook routing The ONLY way to change payouts. Two types: recalc (calc changed — updates SPP.total_payout_usd, preserves original_payout_usd as baseline) and payment_correction (calc was right, payment short — SPP unchanged, top-up only). Each adjustment becomes a payout_adjustments row + a [ADJUSTMENT] ClickUp task tagged with the Adjustment ID custom field. The n8n webhook reads that field, fetches payout_adjustments instead of student_period_payouts, and verifies the Wise transfer against the row’s difference. Each adjustment has its own status/wise_* lifecycle independent of the SPP row.
Visibility
Anyone can see current state without asking		 Ops Dashboard /payouts — period P&L, drill to student → brand → ad. /invoices — brand invoice lifecycle. /students — payment info edits (Ghed’s self-service). /activity — audit log feed. Hosted at ad-bounty-dashboard.vercel.app.

Known Gaps — What We Can’t Yet Catch

Direct-write bypass
If someone sets payout_periods.status = 'complete' directly in Supabase (as happened to April), Skill 8 verification never runs. No trigger yet enforces audit-on-write. Mitigation: only Zak has service-role access; recovery is to re-run finalize after backfilling references.
No second-source on receivables
Brand invoice “paid” status currently trusts Ghed’s data entry. Wise has an API for incoming transfers too; future enhancement is to cross-reference the way we do for outgoing payments.
Wise reference must exist before close
If Ghed marks a ClickUp task paid without pasting the Wise transfer reference, Skill 8 will fail verification for that student (no reference → can’t lookup). This is by design — reference is mandatory.
April 2026 has no verification trail
Period was auto-approved without running Skill 8. Wise references exist in ClickUp custom fields but were never written to student_period_payouts. Brand invoices marked verified with NULL payment data. Backfill plan in progress.
