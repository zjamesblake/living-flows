Integrity Layer Supabase-as-Source

Every state change writes Supabase first, then mirrors to ClickUp. The State Transition Guard (workflow G) watches every ClickUp status change — if ClickUp drifts from Supabase, the guard reverts ClickUp and posts a comment explaining why.

Operator rule: never set ClickUp status manually. Always click the buttons (Approve, Request Revisions, Ready to Launch, Winner, Loser). Manual drags get reverted by the guard.

Trigger-coupled workflows (5.0 Grading, 6.0 Ready to Launch) also bail on bypass — they read Supabase first and skip side-effects if state doesn't match expected, so a manual drag-and-revert leaves no DMs/comments behind.

Why this exists: ClickUp triggers fire on manual drags. Without a guard, a drag-to-approved would send DMs, post grade comments, and corrupt timestamps. The guard makes Supabase the unambiguous source of truth.
External-System Write Standards
Spec Every write across the pipeline follows one of four templates

Every Supabase write, ClickUp status set, ClickUp comment, and Discord DM in the QA pipeline conforms to a canonical template — same auth, same headers, same body shape, same naming. Debugging any write is a five-second read.

  • Supabase: hardcoded service-role JWT, four headers in fixed order (apikey, Authorization, Content-Type, Prefer: return=minimal), state-guard idempotency filter, body keys alphabetized with explicit nulls.
  • ClickUp status: lowercase status strings, written after Supabase, retry on fail (max 2, 2s wait), naming Set Status: <Name>.
  • ClickUp comment: three sources (catalog / code-composed / static-transient), no user tags, naming Comment: <Intent>.
  • Discord DM: canonical embed schema, 4-color palette (green/yellow/red/blue), naming DM: <Intent>.

Quality Review

1. What breaks?

1.0v2 fails on submission: No ClickUp task, no DM. Signal: #ops-digest silence. Student can re-submit safely.
2.0v2 / 3.0v2 Gemini call fails: Task stuck in technical qa / brand qa. No bad data written (return=minimal + idempotency filter). Re-trigger from n8n.
4.2 DM fails: Student doesn't know what to fix. Comment on task still posts; Karlo can relay via ClickUp.
5.0 grading fails after guard pass: Approved but ungraded. Creator DM not sent. Re-trigger 5.0.
6.0 RTL DM fails: Media buyer not notified but Supabase state correct. Re-trigger or notify manually.
7.0 post-launch sync fails: ClickUp lifecycle moves but Supabase lags. Dashboards drift until guard catches it or 7.0 re-runs.
Guard (G) down: Manual drags would corrupt state. Mitigation: train operators that buttons are the only path; drags = no-op.

2. Who knows?

#ops-digest is the heartbeat — silence on a workflow = failure. Check n8n executions.
QA results + override_decision visible on ClickUp tasks and Ops Dashboard. Per-gate precision + FN% tracked in the monthly calibration digest.
Guard reverts post a Comment: Guard Reverted on the task — operator sees the explanation immediately.
Bail-on-bypass events post a Comment: Bypass Detected — auditable trail of skipped side-effects.

3. What's the fallback?

Any workflow down: Supabase remains the source of truth. State can be hand-set via SQL; ClickUp catches up when the guard or 7.0 next runs.
Gemini API down: All QA pauses. No bad data — submissions sit in their current Supabase state. Resume on API return.
Discord bot down: DMs delayed; pipeline keeps running. All results still recorded in Supabase + ClickUp comments. Karlo can relay manually.
QA gates miscalibrated: Monthly calibration digest (per-gate precision, FN-pattern) feeds prompt adjustments. Max 2 rounds → re-do brand interview if still off.
